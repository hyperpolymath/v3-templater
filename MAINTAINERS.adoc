== Maintainers

=== Project Governance

v3-templater follows the *Tri-Perimeter Contribution Framework (TPCF)* for graduated trust and contribution management.

[[current-perimeter-perimeter-3-community-sandbox-earth_africa]]
=== Current Perimeter: *Perimeter 3 (Community Sandbox)* 🌍

This project is in *fully open contribution* mode:

* ✅ *Anyone can contribute* via pull requests
* ✅ *Public issue tracking* - all discussions visible
* ✅ *Community-driven roadmap*
* ✅ *Open code review* - all reviews are public
* ✅ *Transparent decision-making*

==== What This Means

*For Contributors:*

* Fork the repository and submit PRs
* All contributions reviewed by maintainers
* No special access required to participate
* Merit-based advancement to maintainer role

*For Users:*

* Full visibility into development process
* Can report bugs and request features
* Can review ongoing discussions
* Can verify security through public audits

=== TPCF Perimeter Definitions

[[perimeter-3-community-sandbox-current-earth_africa]]
==== Perimeter 3: Community Sandbox (Current) 🌍

* *Access*: Public GitHub repository
* *Contribution*: Anyone can submit PRs
* *Review*: Public code review process
* *Visibility*: All issues, discussions, roadmap public
* *Trust Level*: Open community governance

[[perimeter-2-verified-collaborators-lock]]
==== Perimeter 2: Verified Collaborators 🔒

_(Not currently active)_

* *Access*: Invitation-only collaborators
* *Contribution*: Direct commit access with review
* *Review*: Maintainer approval required
* *Visibility*: Some discussions may be private
* *Trust Level*: Established contributors

[[perimeter-1-core-team-shield]]
==== Perimeter 1: Core Team 🛡️

_(Not currently active)_

* *Access*: Core maintainer team
* *Contribution*: Direct merge authority
* *Review*: Post-commit review acceptable
* *Visibility*: Security-sensitive discussions private
* *Trust Level*: Highest trust level

=== Becoming a Maintainer

==== Path to Maintainer Status

[arabic]
. *Start Contributing* (Perimeter 3)
* Submit quality PRs
* Help with issue triage
* Review others' PRs
* Improve documentation
. *Become Regular Contributor* (~3 months)
* Consistent high-quality contributions
* Deep understanding of codebase
* Helpful to community
* 10{plus} merged PRs
. *Invitation to Perimeter 2* (~6 months)
* Demonstrated expertise
* Trusted code review skills
* Active community participation
* Maintainer nomination
. *Core Team Invitation* (~12 months)
* Sustained contribution excellence
* Leadership in project direction
* Security mindset demonstrated
* Unanimous maintainer approval

==== Maintainer Responsibilities

* *Code Review*: Review PRs within 7 days
* *Issue Triage*: Respond to issues within 48 hours
* *Security*: Follow responsible disclosure
* *Community*: Enforce Code of Conduct
* *Documentation*: Keep docs current
* *Releases*: Participate in release process

=== Decision-Making Process

==== Consensus-Seeking

[arabic]
. *Minor Changes*: Single maintainer approval
. *Feature Additions*: Two maintainer approvals
. *Breaking Changes*: All maintainers consensus
. *Perimeter Changes*: Unanimous core team vote

==== Conflict Resolution

[arabic]
. Discussion in GitHub Issues/PRs
. If no consensus: Lead Maintainer decision
. For major disputes: Community RFC process

== Responsibilities

Maintainers are responsible for:

* Reviewing and merging pull requests
* Triaging issues and feature requests
* Ensuring code quality and security standards
* Managing releases and versioning
* Upholding the project's code of conduct

== Becoming a Maintainer

Contributors who demonstrate:

* Consistent, high-quality contributions
* Understanding of the project's goals and standards
* Constructive participation in discussions
* Commitment to the project's long-term health

May be invited to become maintainers at the discretion of existing maintainers.

== Decision Making

* Routine decisions (bug fixes, minor improvements) can be made by any maintainer
* Significant changes require discussion and consensus among maintainers
* Breaking changes or major features should be discussed in issues before implementation

== Contact

For questions about project governance, open an issue or contact the maintainers listed above.
